# 项目三：多Agent自我进化系统 — 面试问答集

> 适用岗位：AI Agent 开发 / 大模型应用 / NLP 工程
> 回答格式：每道题先给你"为什么做→具体做法→好处→举一反三"，最后附面试话术模板（直接背的）

---

## 一、项目概览篇（必问）

### Q1：请介绍一下你这个项目

**面试官要什么：** 30 秒内说清"这是什么项目、解决了什么问题、有什么亮点"。

**回答框架：**

**为什么做这个项目：**
RAG 项目做了两轮后发现一个问题——每次改代码都要手动测试、手动审查、手动 Git 提交，这些重复劳动占开发时间 40%。而且**自己审查自己的代码很难发现 bug**。项目三的目标就是让 AI 自己完成这些流程化工作。

**具体是什么：**
9 个 Hermes Agent 分成两队——A 队 4 个负责写代码和建 Skill，B 队 4 个负责审查代码和写测试，1 个协调者负责调度和 Git 提交。每 30 分钟自动跑一轮，零人工干预。

**亮点有哪些：**

| 维度 | 亮点 |
|------|------|
| 自我编码 | Agent 自主选择 TODO 任务并实现 |
| 自我审查 | B 队独立审查 A 队，发现盲区 |
| 自我进化 | 踩过的坑写入 memory，下次不犯 |
| 自我提交 | 自动 git add → commit → push |
| 全部免费 | 本地 Ollama + Qwen2.5，零 API 费用 |

**面试话术（30秒版）：**
> "这是一个让 AI Agent 自我进化的系统。9 个 Agent 分成两队，A 队写代码建 Skill，B 队审查代码保质量，协调者做最终决策和 Git 提交。每半小时自动循环一次，不用人管。设计上参考了大厂的 Code Review 流程——开发团队和 QA 团队分离，避免了自查盲区。"

**面试话术（1分钟版）：**
> "这个项目的出发点是：我之前做 RAG 项目，每次改代码都要手动测试、手动审查、手动 Git 提交，重复性工作很多。所以我设计了一个多 Agent 系统让 AI 自己完成这些事。架构分三部分：A 队 4 个 Agent 负责开发，B 队 4 个 Agent 负责质量保障，协调者做调度和决策。每轮 A 队并行产出代码，B 队并行审查，协调者汇总后 Git 提交，全过程 20 分钟左右。关键设计是两队完全隔离、用文件通信、只有协调者有 Git 权限——这保证了并发安全和数据一致性。"

### Q2：9 个 Agent 为什么分两队？不能一个 Agent 包揽吗？

**面试官要什么：** 考察你对"分工"和"质量控制"的理解，是否知道 Code Review 的价值。

**为什么这样设计：**
一个人写代码自己审查，是很难发现自己写的 bug 的。这就像自己检查自己写的作文——看了五遍还是漏掉那个错别字。**需要外部视角。**

**具体对比：**

| 方案 | 问题 | 风险 |
|------|------|------|
| 单 Agent 全干 | 写→审→提都是同一个人 | 写错了一路错到 Git，没人发现 |
| A+B 两队制 | A队专注产出，B队独立审查 | B队不受A队代码风格影响，客观评价 |

**好处：**
- 内建制衡——A 队不敢乱写，因为知道后面有人查
- 捕获盲区——B 队能看到 A 队忽略的边界情况和异常分支
- 职责分离——Agent 不用记"怎么开发"又要记"怎么审查"，每个 Skill 更专注

**举一反三：**
这和现实中的研发流程一模一样——大厂的每个 PR 必须有至少一个人 Review 才能合并到主分支。GitHub 的 Code Review 功能、Google 的 critique 工具，都是这个逻辑。项目三只是把人换成了 AI。

**面试话术：**
> "分两队是为了内建制衡。单 Agent 做开发又做审查，等于自己判自己的卷子——很难发现自己的 bug。我参考了大厂的 Code Review 流程，让 A 队专注产出、B 队独立审查。A 队是开发者视角，讲的是'先把功能跑起来'；B 队是质量视角，讲的是'边界情况处理了吗、异常捕获了吗、代码风格统一了吗'。两队视角不一样，互相补充才能保证质量。"

### Q3：这个系统和普通的定时脚本（cronjob）有什么区别？

**面试官要什么：** 考察你是否理解"Agent"和"脚本自动化"的本质区别——**Agent 有自主决策能力，脚本只是机械执行。**

**为什么需要这个区别：**
普通 cronjob 是"这行写死、下一行执行"，永远不会变。但项目三要解决的问题是"怎么让系统自己发现要做什么、自己判断做得好不好"——这需要 Agent 有决策和推理能力，不只是执行能力。

**对比表：**

| 维度 | 普通 cronjob 脚本 | Agent 系统 |
|------|------------------|------------|
| 任务来源 | 写死的命令列表 | 自己读 TODO.md 选任务 |
| 执行什么 | `python backup.py` | Agent 理解"这个任务难度匹配吗" |
| 质量判断 | 检查 exit code | Agent 读代码、审逻辑、测功能 |
| 出错处理 | 发通知 | 分析原因、记 memory、下次避免 |
| 进化能力 | 写死，不进化 | 建 Skill、写 memory、自我改进 |

**好处：**
- 系统能处理未知情况——如果 TODO 被清空，协调者自动生成新任务
- 长期运行质量越来越好——第一次写 `write_file` 忘了异常处理，B队发现后记入 memory，下次就记得了
- 不需要人干预维护——脚本挂了你要上去改，Agent 系统自己修自己

**举一反三：**
这就像**自动驾驶**和**定速巡航**的区别。定速巡航只是维持一个固定速度（普通脚本），碰到前车刹车就不行了。自动驾驶能感知环境、做决策、适应变化（Agent 系统）。项目三追求的是后者——不是写死的流水线，是能自主决策的智能体。

**面试话术：**
> "普通 cronjob 是死的，只会跑固定的命令。我的系统是活的——Agent 会读 TODO.md 自己选任务、自己判断任务难度是否匹配、自己写代码并验证。跑崩了也不会丢一边，而是分析原因记入 memory，下次改进。区别核心就是有没有自主决策能力。打个比方：定速巡航和自动驾驶的区别。脚本是定速巡航，系统是自动驾驶。"

## 二、架构设计篇

### Q4：每轮循环具体是怎么跑的？描述完整流程

**面试官要什么：** 考察你对系统流程的熟悉程度——不是背课本，是说清楚每个阶段输入输出。

**完整流程（20 分钟一轮）：**

```
T+0min  协调者启动
         输入：TODO.md + CHANGELOG.md + 项目文件
         输出：4 个 A 队子 Agent 的任务描述
         
T+1min  A队 × 4 并行启动（delegate_task batch）
         输入：TODO 任务 + 项目上下文 + dev-cell 技能
         输出：.py 文件 + SKILL 文件 + report.json
         
T+1~8min 每个 A 队 Agent 独立工作：
         读 TODO → 选任务 → 读项目文件了解上下文
         → 写代码 → 验证语法 → 写 report.json
         先完成的不用等别人，独立输出到 tmp_agent/agent-{id}/
         
T+8min  协调者汇总 A 队产物
         输入：4 份 report.json + 实际产出文件
         输出：4 份审查需求（给 B 队）

T+9min  B队 × 4 并行启动（delegate_task batch）
         输入：A 队产出文件 + qa-cell 技能
         输出：review.json
         
T+9~16min 每个 B 队 Agent 独立审查：
         读产物 → 代码审查/写测试/查文档/查安全
         → 标注严重级别 → 写 review.json

T+16min 协调者汇总审查报告
         输入：4 份 review.json
         输出：决策结果（合并/驳回/修复）

T+17min 决策执行
         合并：保留 A 队产出
         驳回：回滚改动，记录原因到 TODO
         修复：快速指派单个 Agent 修 critical 问题

T+18min Git 提交
         git add -A → git commit -m "round N ..." → git push
         更新 TODO.md（已完成打勾）
         更新 CHANGELOG.md

T+19min 本轮完成，等待下一轮（每 30 分钟触发）
```

**关键技术细节（面试加分）：**

| 机制 | 为什么这么设计 |
|------|---------------|
| 文件通信 | 子 Agent 不能共享变量，通过 `tmp_agent/{agent_id}/` 目录传递 JSON 报告 |
| batch 并行 | `delegate_task` 的 batch 模式让 4 人同时开工，节省 60% 时间 |
| 语法验证 | 每个 Agent 写完必须 `python -c "import ast; ast.parse(...)"` 验语法 |
| 只有协调者能 Git | 8 个 Agent 同时 `git push` 会乱套，单一提交源保证原子性 |

**面试话术：**
> "一轮循环分四个阶段。第一阶段 A 队 4 个 Agent 并行开发，每人从 TODO 选一个任务，写完验语法后写 report 到临时目录。第二阶段 B 队 4 个 Agent 并行审查，检查代码质量、写测试、看文档是否完整。第三阶段协调者汇总审查报告，做决策——没大问题就合并，有致命问题就驳回。第四阶段执行 Git 提交和更新 TODO。整个过程约 20 分钟，通过 cronjob 每 30 分钟触发一次，留 10 分钟缓冲。"

### Q5：A 队 4 个 Agent 怎么分工？不会重复做一样的事吗？

**面试官要什么：** 考察并行调度设计中的任务分配策略和冲突避免。

**分工设计：**

| Agent | 角色 | 关注点 | 不会做的事 |
|-------|------|--------|-----------|
| Agent 1 (dev-core) | 核心逻辑 | 写业务函数、算法 | 不改配置、不改文档 |
| Agent 2 (dev-tools) | 工具/接口 | 写工具函数、CLI | 业务逻辑、Skill 创建 |
| Agent 3 (dev-skills) | 知识/Skill | 写 SKILL、README、文档 | 不写 Python 代码 |
| Agent 4 (dev-memory) | 记忆/配置 | 分析 TODO、管理配置 | 不创建文件 |

**为什么不会重复：**

| 防冲突机制 | 说明 |
|------------|------|
| 职责范围固定 | 每个 Agent 的 Skill 技能描述中明确了"你做什么、不做什么" |
| 协调者统一派发 | 协调者读出 TODO 后，分配给最匹配的 Agent |
| 文件锁定 | 如果一个文件已被创建，B 队审查前其他 Agent 不碰 |
| 协调者仲裁 | 如果真的冲突了，协调者保留一个版本，另一个驳回 |

**好处：**
- 4 个 Agent 可以同时开工，互不抢文件
- 每个 Agent 的上下文小（只关注自己领域），推理更准确
- 单个 Agent 失败不影响其他人

**面试话术：**
> "每个 Agent 有严格的分工范围。Agent 1 专注写算法逻辑，Agent 2 写工具函数，Agent 3 写文档和 SKILL，Agent 4 管记忆和配置。技能描述里明确写了'你负责什么、不管什么'，所以不会做重复的事。如果万一有重叠，协调者做最终裁决。这样设计的好处是 4 人可以并行开工、上下文隔离、单个挂了不影响整体。"

### Q6：A 队产出代码质量不高怎么办？B 队漏审查了怎么办？

**面试官要什么：** 考察容错设计和对"自愈系统"的理解。

**A 队质量不高的处理：**

```
A队代码质量低
  ↓
B队审查发现（critical / major）
  ↓
协调者决策：
   ├─ 可修复 → 派快速修复（再跑一个子Agent改）
   └─ 不可修 → 驳回，本轮跳过，TODO 保留
  ↓
问题记入 memory：
  "上次 test_xx.py 断言写法不对 → 下次 B队重点检查断言"
  ↓
下一轮质量自动提升
```

**B 队漏审查的处理：**

```
如果 B 队漏了 critical 问题：
  ↓
代码合并到 Git → 下一轮 A 队用这个代码继续开发
  ↓
新的 A 队代码依赖了有 Bug 的基础 → 测试跑不过
  ↓
B 队回头看 → 发现是上一轮的基础代码就有问题
  ↓
协调者决策：本轮修正上轮的 Bug + 记入 memory
  ↓
"检查标准不严格 → 下轮加强审查力度"
```

**关键设计：漏审查不会永久性伤害系统**

| 轮次 | 状态 |
|------|------|
| Round N | A 队写了 Bug，B 队没发现，合并了 |
| Round N+1 | 新代码依赖了这个 Bug，测试失败 |
| Round N+2 | 系统修复了 Bug，改进了审查标准 |
| Round N+3+ | 不再犯同类错误 |

**好处：**
- 系统有"纠错余量"——不是一轮定生死
- 漏审一次不可怕，可怕的是**系统不记住**。memory 就是"不忘记"

**面试话术：**
> "质量问题的核心不是'不出错'——是人都会出错，Agent 也会。关键是怎么兜底。我设计了三个保障：第一，A 队代码必须过语法验证，基础错误跑不掉。第二，B 队 4 人从不同维度审查，代码、测试、文档、安全，覆盖全面。第三，就算漏了，下一轮新代码跑了之后的测试会发现上轮的问题，形成自愈闭环。再配合 memory 记录错误模式，系统质量是螺旋上升的，不会原地踏步。"

## 三、技术实现篇

### Q7：你们用的是什么模型？为什么选本地模型？

**面试官要什么：** 考察技术选型能力和对"成本 vs 效果"的权衡。

**选型方案：**

```
Ollama + Qwen2.5-7B-Instruct (Q4_K_M 量化)
           ↓
本地运行，不联网 → 完全免费、完全离线
           ↓
RTX4060 8GB 显存
~4.5GB 给模型，剩余 3.5GB 给系统和 Hermes
           ↓
所有 Agent 共用同一个模型实例
```

**为什么不用商业 API：**

| 方案 | 成本（30天×48轮） | 延迟 | 离线 | 可用性 |
|------|-------------------|------|------|--------|
| Ollama 本地 | 0 元 | 8-15 tok/s | ✅ | 100% |
| DeepSeek API | ~50-200 元 | 快 | ❌ | 99.5%+网络 |
| OpenAI API | ~200-800 元 | 快 | ❌ | 99.9%+网络 |

**为什么选 7B 而不是 3B 或 14B：**

| 模型 | 显存 | 代码质量 | 选择原因 |
|------|------|----------|----------|
| 3B | ~2GB | 一般 | 不够，写复杂逻辑经常出错 |
| **7B Q4** | **~4.5GB** | **中等** | **平衡点——质量够 + 显存余量够** |
| 14B Q4 | ~9GB | 好 | 显存放不下，4060只有8GB |

**代价：**
7B 模型写代码质量不如 GPT-4——所以 B 队的审查就更重要。A 队写的一般，B 队把关兜底。这是设计上主动接受的 trade-off。

**面试话术：**
> "我选了 Ollama + Qwen2.5-7B 本地运行。原因很简单——这个系统每 30 分钟跑一轮，一个月 1440 轮，如果用 OpenAI API 那成本根本扛不住。本地模型零成本、零网络依赖，7B Q4 量化在 4060 上够跑。代价是代码质量不如 GPT-4，但没关系——B 队的审查就是为了兜底这个质量差距。选型本质是算账：1440 轮 vs 0 元 vs 可接受的质量，本地模型赢了。"

### Q8：子 Agent 之间用文件通信，为什么不直接用共享变量或数据库？

**面试官要什么：** 考察通信协议设计——为什么选文件而不是内存/数据库。

**对比方案：**

| 方案 | 复杂度 | 并发安全 | 持久化 | 实现成本 |
|------|--------|----------|--------|----------|
| **文件 JSON** | 低 | ✅（写不同文件） | ✅ | 0 |
| 共享变量/内存 | 高 | ❌（并发写冲突） | ❌ | 中 |
| 数据库（SQLite） | 中 | ✅（事务） | ✅ | 高 |
| Redis | 高 | ✅ | ⚠️ | 高 |

**为什么选文件：**

1. **子 Agent 是隔离的——** 每个子 Agent 只写自己的目录 `tmp_agent/agent-{id}/`，不会写到别人的地盘。天然无冲突。
2. **文件是 JSON——** 协调者 `json.load()` 就读进来了，不需要安装 Redis、不需要建表。
3. **文件是持久的——** 如果系统崩了，`tmp_agent/` 里的报告可以作为复盘材料。
4. **零依赖——** 不需要 MySQL、不需要 Redis、不需要任何中间件。项目三的依赖列表只有一个 `pytest`。

**什么情况下会换方案：**
如果系统迭代到 100+ Agent 或 秒级调度，文件 IO 会成为瓶颈。到那时会换成 Redis Stream 或 RabbitMQ。但当前 9 个 Agent、20 分钟一轮，文件通信完全够用。

**举一反三：**
这和选择**结构化日志**而不是数据库做系统监控是一个逻辑。日志文件简单、可靠、零依赖，80% 的场景下比数据库更合适。

**面试话术：**
> "我选了最朴素的文件通信。不是因为 Redis 不好，是没必要——每个子 Agent 只写自己的目录，天然不会冲突；协调者一个 json.load() 就读回来了。文件方案零依赖、零安装、持久化，适合这个规模的需求。等以后扩展到大几十个 Agent 再考虑换中间件。这个设计原则就是：用简单的工具解决 80% 的问题，复杂工具留给够不着的 20%。"

### Q9：怎么防止 Agent 陷入死循环或无限重复？

**面试官要什么：** 考察**系统稳定性设计**——能不能提前想到故障模式。

**四种死循环及对策：**

| 死循环类型 | 场景 | 防护 |
|-----------|------|------|
| **TODO 为空** | 所有任务都完成了，Agent 不知道该干嘛 | 协调者检测到空 TODO → 自动生成"破局任务"（检查质量/重构/优化） |
| **B 队永不通过** | 每轮都驳回，A 队改了一万次 | 协调者有强制合并权，连续驳回 3 轮后自动通过 |
| **Git 永远无变化** | A 队写了觉得不好又删了，equality check 没变化 | 协调者检测到连续 5 轮无变化 → 主动生成大任务 |
| **子 Agent 全部失败** | 模型卡了或网络断了 | 协调者跳过本轮，记录原因到 CHANGELOG，下一轮自动恢复 |

**超时机制：**
每个阶段都有最大等待时间——A 队最长 12 分钟、B 队最长 12 分钟。超时未返回的 Agent 记为失败，不阻塞整体流程。

**面试话术：**
> "我预设了四种最可能的死循环场景并做了防护。TODO 空了就自动补任务、B 队卡审就有强制合并权、Agent 没产出就超时跳过、Git 没变化就生成大任务。核心思想是：系统绝不能死在某个阶段卡住不走了——宁可跳过这一轮，也要保证下一轮能正常启动。"

## 四、比较与辨析篇

### Q10：你这个系统和 AutoGPT / LangChain 的 Multi-Agent CrewAI 有什么区别？

**面试官要什么：** 考察你对同类技术的了解深度，有没有做过调研。

| 维度 | AutoGPT/CrewAI | 我的项目三 |
|------|---------------|-----------|
| 目标 | 一次完成一个用户指定的任务 | **永久运行，自我进化** |
| 学习能力 | 每个会话独立，不会积累经验 | memory + SKILL 跨会话积累 |
| 审查机制 | 无（单个Agent自己干活） | 两队制衡，有人审 |
| 持久化 | 无（输出即结束） | Git 版本控制，所有改动有记录 |
| 成本 | 通常依赖 OpenAI API | 本地模型，免费 |

**核心区别一句话：**
AutoGPT 是"帮我完成一个任务，做完就结束"。项目三是"帮我维护一个代码库，长期持续迭代"。

**为什么不用 CrewAI：**
- CrewAI 的 Agent 是"一次对话、一次任务"，没有跨轮记忆
- CrewAI 没有内置的审查制衡，没有质量门禁
- CrewAI 依赖 LangChain 生态，增加复杂度
- **我的需求不复杂——9 个 Agent、20 分钟、10 个文件**，不需要大框架

**面试话术：**
> "AutoGPT 和 CrewAI 是任务型的——你给一个目标，它做完就完了。项目三是持续型的——它要一直运行、越跑越好。而且 AutoGPT 依赖 OpenAI API，我的系统全本地跑。CrewAI 也缺少审查制衡，我的系统有 B 队把关。不是谁好谁坏，是定位不同——他们是'一次性智能助手'，我做的是'长期运营的 AI 开发者'。"

### Q11：做这个项目踩过什么坑？怎么解决的？

**面试官要什么：** 考察**实战经验**——背书的候选人和踩过坑的候选人有本质区别。

| 坑 | 问题 | 解决方案 |
|----|------|----------|
| **子 Agent 冲突** | 两个 Agent 同时编辑 `README.md`，互相覆盖 | 统一由协调者分配文件范围，不重叠 |
| **模型太慢** | Qwen2.5 7B 每轮推理要跑很久 | 降低调度频率到45分钟 |
| **Git commit 重复** | 如果本轮没新内容，commit 会报"nothing to commit" | 先 `git add` → 检查 `git diff --cached --quiet` → 没变化就不 commit |
| **子 Agent context 太大** | 把 TODO 全文传给子 Agent 浪费 token | 只传和该 Agent 角色相关的 TODO 子集 |

**面试话术：**
> "踩了四个坑。第一是文件冲突——两个 Agent 同时改一个文件，互相覆盖。我改成统一由协调者分配文件范围，不重叠。第二是本地模型慢——7B 推理要几十秒，我把调度频率从 10 分钟调到 30 分钟。第三是 Git 重复提交——先检查有没有变化再 commit。第四是 context 浪费——只传和该 Agent 角色相关的 TODO。这些问题都是上线跑起来才发现的，理论设计和实际运行总有落差，跑几轮就能发现问题。"
